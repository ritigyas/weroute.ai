# Weroute.ai – Network Device Health Platform

---

## 1. Overview & Business Objective

**Purpose**: Provide network operators a single pane‑of‑glass to monitor the health, configuration, and performance of a fleet of network devices (primarily Juniper).  The platform runs periodic **NETCONF** data collection via Ansible, stores per‑device JSON snapshots, and exposes a **FastAPI** service that powers a **Streamlit** dashboard.  An on‑demand **ICMP ping** job enables quick reachability checks without re‑running the full NETCONF collection.

**Target Users**: NOC engineers, network reliability teams, and automation developers who need to:
- Detect device‑level faults (BGP down, interface failures, environmental alerts).
- Visualise fleet‑wide health trends.
- Run ad‑hoc reachability tests.
- Extend the platform with custom Ansible playbooks.

---

## 3. Component Responsibilities

| Component | Responsibility |
|-----------|-----------------|
| `main.py` | FastAPI entry point; defines routes, CORS, and background task handling. |
| `collector.py` | Creates **CollectJob** and **PingJob** objects, launches asynchronous `ansible-playbook` processes, captures logs, assigns a unique **run_id** (compact UTC timestamp). |
| `store.py` | Reads JSON result files from `data/results/`, normalises them via `junos_parse.build_health`, and surfaces `DeviceHealth` and `FleetSummary` Pydantic models. |
| `models.py` | Pydantic data models (`DeviceHealth`, `CollectJob`, `PingJob`, `FleetSummary`, etc.). |
| `collect.yml` | Netconf collection playbook – gathers facts, BGP, interfaces, environment, uptime; writes a JSON file per device including the supplied `run_id`. |
| `ping.yml` | Minimal playbook that runs Ansible `ping` module on each target, emitting `PING_RESULT` lines parsed by the backend. |
| `dashboard.py` | Streamlit UI – sidebar to trigger collection/ping, auto‑refresh, displays fleet summary cards, device table, detailed expanders, and ping results. |
| `store.py` + `junos_parse.py` (not shown) | Transform raw Ansible JSON into the unified health schema. |

## 4. Data Model & Storage Strategy

### 4.1 JSON Record Layout (per device)
```json
{
  "hostname": "spine1",
  "address": "10.0.0.1",
  "group": "spine",
  "timestamp": "2024-01-15T10:30:00Z",
  "run_id": "20240115T103000Z",
  "reachable": true,
  "netconf_ok": true,
  "ping_rtt_ms": 1.23,
  "facts": {
    "model": "MX480",
    "version": "21.3R3",
    "serial": "ABCD1234"
  },
  "raw": {
    "route_engine": {...},
    "bgp_summary": {...},
    "interfaces": {...},
    "environment": {...},
    "uptime": {...}
  },
  "last_ping_at": "2024-01-15T11:00:00Z",
  "last_ping_rtt_ms": 1.23,
  "last_ping_reachable": true
}
```

### 4.2 Directory Layout
```
weroute.ai/
├─ ansible/                # playbooks
├─ data/
│   └─ results/            # JSON files per device (generated by playbooks)
├─ collector.py            # job orchestration
├─ dashboard.py             # Streamlit UI
├─ main.py                  # FastAPI service
├─ models.py                # Pydantic schemas
└─ store.py                 # persistence layer
```

### 4.3 Run‑ID Mechanism & Retention
- Every collection run receives a **run_id** (`YYYYMMDDTHHMMSSZ`).
- The playbook stamps this `run_id` into each JSON file.
- After a successful run, `collector._purge_old_runs` deletes any JSON whose `run_id` differs from the current one, ensuring **only the latest snapshot** remains on disk.
- Ping jobs **patch** only `last_ping_*` fields; they never trigger a purge.

## 5. API Endpoints (FastAPI)

| Method | Path | Description | Request Params | Response Model |
|--------|------|-------------|----------------|---------------|
| `GET` | `/api/health` | Liveness probe | – | `{status: "ok"}` |
| `GET` | `/api/summary` | Fleet‑wide health aggregation | – | `FleetSummary` |
| `GET` | `/api/devices` | List all device health records | – | `List[DeviceHealth]` |
| `GET` | `/api/devices/{name}` | Single device details | `name` path | `DeviceHealth` |
| `POST`| `/api/collect` | Trigger full NETCONF collection (background) | `target` optional query | `CollectJob` (id, status, run_id, etc.) |
| `GET` | `/api/jobs/{job_id}` | Poll collect job status | `job_id` path | `CollectJob` |
| `POST`| `/api/ping` | Trigger on‑demand ICMP ping sweep | `target` optional query | `PingJob` |
| `GET` | `/api/ping-jobs/{job_id}` | Poll ping job status + per‑host results | `job_id` path | `PingJob` |

**Execution flow** for `/api/collect`:
1. `create_job` stores a `CollectJob` in the in‑memory `JOBS` dict (status = queued). 
2. FastAPI adds `run_collection` as a background task. 
3. `run_collection` generates a new `run_id`, invokes `ansible-playbook collect.yml` with `-e run_id=…`. 
4. Ansible writes JSON files stamped with that `run_id`. 
5. On success, `_purge_old_runs` removes previous snapshots. 
6. Job object is updated (status, returncode, log tail, deleted_runs). 

The `/api/ping` flow mirrors the above but uses `PING_JOBS`, runs `ping.yml`, parses `PING_RESULT` lines, and **patches** the existing JSON files (only ping fields). No purge occurs.

## 6. Runtime Walkthrough (Story)

1. **Application Startup**
   - `uvicorn` launches `main.py`; FastAPI creates the `app` object, registers CORS, and mounts all routes.
   - Global dictionaries `JOBS` and `PING_JOBS` are empty. `DATA_DIR` points to `data/results/`.

2. **User Opens Dashboard**
   - Streamlit runs `dashboard.py`. It queries `/api/summary` and `/api/devices` to populate the fleet cards and device table.
   - All status emojis map to device health (`healthy`, `warning`, `critical`, `unknown`).

3. **Trigger Full Collection** (sidebar **Collect** button)
   - UI POSTs to `/api/collect` (optional `target`).
   - FastAPI creates a `CollectJob` (`status=queued`) and schedules `run_collection`.
   - `run_collection` sets `status=running`, creates a compact UTC `run_id`, and spawns `ansible-playbook collect.yml`.
   - Playbook runs per‑device tasks: ICMP reachability, NETCONF fact collection, BGP, interfaces, environment, uptime.
   - At the end, each device writes a JSON file containing all gathered data plus the `run_id`.
   - Backend receives the process exit code, stores the tail of stdout for debugging, marks job `success` or `failed`, and calls `_purge_old_runs` to delete stale snapshots.
   - UI polls `/api/jobs/{id}`; once `status=success`, the dashboard reloads data, showing the new snapshot and the **Run ID** label.

4. **Run Ping‑Only Sweep** (sidebar **Ping devices** button)
   - UI POSTs to `/api/ping`.
   - FastAPI creates a `PingJob` and schedules `run_ping`.
   - `run_ping` executes `ansible-playbook ping.yml`. The playbook prints lines like `PING_RESULT host=spine1 ...`.
   - Backend parses these lines (`_parse_ping_output`) into a list of `PingResult` objects.
   - `_patch_ping_results` opens each device’s JSON file and updates `last_ping_*` fields, leaving all NETCONF data untouched.
   - UI polls `/api/ping-jobs/{id}`; results appear in an expandable table and update the **Last ping** column of the main device table.

5. **Dashboard Refresh**
   - Auto‑refresh (15 s) or manual **Refresh** button re‑issues the `/api/summary` and `/api/devices` calls, pulling the latest snapshot and any recent ping data.

## 7. Health Calculation Logic (`junos_parse.build_health`)
*(Implementation lives in `junos_parse.py` – omitted for brevity)*
1. Extracts **BGP peer** list, counts total and down peers.
2. Derives interface totals, admin‑up vs oper‑up counts, and builds a list of down interfaces.
3. Reads environment items, counting total sensors and those not `ok`.
4. Computes **status**:
   - `critical` if any BGP down peers, interface down, or environment alerts.
   - `warning` if reachable but some non‑critical issues exist.
   - `healthy` if reachable and all subsystems report `ok`.
   - `unknown` if device not reachable.
5. Populates `alerts` with human‑readable messages (e.g., "BGP peer X down", "Interface ge‑0/0/0 oper down").

## 8. Dashboard UI Details
- **Sidebar**: API base URL, optional target filter, **Collect**, **Ping**, auto‑refresh toggle.
- **Fleet summary cards**: total devices, reachable, healthy, warning, critical.
- **Device table**: status emoji, hostname, IP, model/version, NETCONF reachability, ping RTT, CPU, memory, BGP down count, interface down count, last‑ping columns, alert count.
- **Device detail expander**: uptime, CPU, memory, temperature, latest ping info, BGP peer table, interface summary with down list, environment items table.
- **Ping results expander**: tabular view of per‑host ping outcome after a ping job.
- **Auto‑refresh** loops with `time.sleep(15)` and `st.rerun()`.

## 9. Developer On‑boarding Guide
### 9.1 Prerequisites
| Tool | Version |
|------|---------|
| Python | >=3.10 |
| pip   | latest |
| Ansible Core | >=2.15 |
| `junipernetworks.junos` collection | latest |
| FastAPI | >=0.115 |
| Uvicorn | >=0.30 |
| Streamlit | >=1.38 |
| `pydantic` | >=2.7 |

### 9.2 Installation
```bash
# Clone the repo
git clone https://github.com/yourorg/weroute.ai.git
cd weroute.ai

# Create a virtual environment
python -m venv .venv
source .venv/Scripts/activate   # Windows PowerShell

# Install dependencies
pip install -r requirements.txt
```
> *If `requirements.txt` is missing, install manually:* `pip install fastapi uvicorn streamlit pydantic ansible ansible-core junipernetworks.junos`

### 9.3 Configuration
- Optionally set environment variables (default values shown):
```bash
# Directory where Ansible playbooks live (relative to project root)
export ANSIBLE_DIR=$(pwd)/ansible
# JSON results storage location
export RESULTS_DIR=$(pwd)/data/results
# API base URL for the dashboard (default localhost)
export API_BASE=http://localhost:8000
```
- No additional config files are required.

### 9.4 Running the Services
```bash
# Start FastAPI backend (development mode)
uvicorn main:app --reload --port 8000
```
In another terminal:
```bash
# Launch the Streamlit UI
streamlit run dashboard.py
```
The dashboard will be reachable at `http://localhost:8501`.

### 9.5 Using the API
- **Collect**: `POST http://localhost:8000/api/collect?target=spine1`
- **Check job**: `GET http://localhost:8000/api/jobs/{job_id}`
- **Ping**: `POST http://localhost:8000/api/ping`
- **Check ping job**: `GET http://localhost:8000/api/ping-jobs/{job_id}`
- **List devices**: `GET http://localhost:8000/api/devices`
- **Summary**: `GET http://localhost:8000/api/summary`

### 9.6 Extending / Adding New Playbooks
1. Add a new YAML playbook under `ansible/`.
2. Create a corresponding endpoint in `main.py` (similar to collect/ping) that spawns a background task calling a helper in `collector.py`.
3. Update `store.py` if the new playbook writes additional fields – ensure `build_health` extracts them.
4. Adjust the Streamlit UI to visualise any new data (add columns, tabs, or cards).

### 9.7 Maintenance & Scalability
- **Storage**: Only the latest snapshot is kept; each device file is < 50 KB, so a fleet of 5 000 devices uses ~250 MB.
- **Background Tasks**: FastAPI uses Starlette’s background task system; for larger fleets consider moving to a task queue (Celery/RQ) and a dedicated worker pool.
- **Parallel Ansible**: Add `forks` configuration in `ansible.cfg` to increase simultaneous device collection.
- **Health Calculation**: The current algorithm is O(N) per device; if more complex analytics are needed, off‑load to a separate analytics service or a database.

### 9.8 Security Considerations
- **API Exposure**: In production, restrict CORS origins, enable HTTPS (via a reverse proxy), and add authentication (OAuth2/JWT). 
- **Credentials**: Ansible inventory should reference encrypted vault files; never store clear‑text passwords in the repo.
- **File Permissions**: Ensure `RESULTS_DIR` is readable only by the service user.
- **Input Validation**: Both `target` query parameters are passed directly to Ansible; validate against an allow‑list of known host/group names to avoid injection.

### 9.9 Troubleshooting Checklist
| Symptom | Likely Cause | Debug Steps |
|----------|--------------|-------------|
| No devices appear | `RESULTS_DIR` empty or mismatched path | Verify `RESULTS_DIR` env var, inspect `data/results/`.
| Collection job stays *running* forever | Ansible playbook hanging or unreachable device | Check the background task log (`job.log_tail`), run the playbook manually: `ansible-playbook collect.yml -e run_id=test`.
| Ping job returns *failed* | Missing `ansible` binary or wrong playbook path | Ensure `ansible-playbook` is on PATH; run `ansible-playbook --version`.
| Dashboard shows stale run‑id | `_purge_old_runs` not executed (e.g., collection failed) | Examine `job.deleted_runs` field; re‑run a successful collection.
| UI errors on refresh | API unreachable | Verify backend is running on the configured port.
